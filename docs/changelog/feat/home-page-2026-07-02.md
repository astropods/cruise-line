# Public home page for CLI download and per-user token management

## Summary

The CLI-tokens section had been living inside `/settings`, which turned that
page into a grab bag of unrelated concerns and — worse — put a per-user
feature behind an owner-only guard. Non-owner collaborators visiting
`/settings` hit a "not authorized" dead end, so an earlier fix re-embedded
the tokens section inside `LockedScreen` as a workaround.

This change gives the CLI its own home. `/` is now the public entry point
for anyone downloading the CLI or managing their tokens; `/settings` goes
back to being purely install-level administration.

## Design

### Route separation

The home page is deliberately outside the setup guard that fronts every
other route. Anyone with the URL can see the install one-liner without
authenticating — which is exactly what enables `curl | sh` bootstrap and
lets a coworker share the URL as "here's how to install our CLI." Signed-in
extras (identity, token list, owner nav) render conditionally on the same
page.

`/settings` narrows to install-level admin (repositories, users, owner
role). Non-owner users who arrive there are redirected to `/` rather than
shown a locked-out screen — everything they can do lives on the home page,
so a redirect is a cleaner outcome than a "you cannot see this" message
followed by exactly the section they *can* see.

### Composition

`CliTokensSection` moved from being a private component inside
`SettingsPage.tsx` to `components/CliTokensSection.tsx`. It's now a
freestanding React component with its own state, ready to be dropped
wherever a "manage the current user's tokens" surface is needed. The home
page mounts it inline; nothing else does today.

The home page itself is composed of three numbered sections — install,
sign in + issue token, try it — plus a top nav and a footer. Copyable
code blocks handle both the install command and the getting-started
snippets. The install command's host is fetched from `/api/setup/status`
(the same public endpoint used elsewhere for the app URL) so the string
shown in the browser is always the real deploy URL, not a placeholder.

```
GET /api/setup/status → { appUrl: "https://cl.example.com", ... }
```

That single fetch drives the install one-liner:

```
curl -fsSL https://cl.example.com/install.sh | sh
```

### Auth-optional data loading

The home page needs to detect "signed in vs not" without forcing the
`fetchUser` helper's redirect-on-401 behavior. The page calls
`/api/auth/me` directly and treats a 401 as a normal "signed out" state.
The result gates the visible content — anonymous visitors see the install
command and a "Sign in with GitHub" button; signed-in users see their
identity, a `cruise-line login <host>` command, and their token list.

Owners see one extra affordance: a subtle "Install settings →" link in the
footer, and a "Finish setup" button in the top nav when the install is
unconfigured. Non-owners see nothing at all pointing at `/settings`.

## Migration

No migration required. All routing changes are additive — existing URLs
(`/settings`, `/cli/authorize`, `/:owner/:repo/pull/:pr`) still work the
same way. Non-owner users who had `/settings` bookmarked will now get
redirected to `/`, which is where their tokens actually live.
